# Writing Guidelines

Guidelines for UX writing in Apple platform apps — voice, tone, clarity, and content patterns.

## Establishing Voice & Tone

- **Determine your app's voice.** Create a list of common terms and reference it for consistency. Consistent language reflecting your app's values makes everything feel cohesive.
- **Match tone to context.** Vary tone based on the situation — serious for fall detection, celebratory for achievements. Consider what people are doing both in the physical world and within the app.

## Core Principles

- **Be clear.** Choose easily understood words. Remove unnecessary words. If fewer words work, use fewer. Read writing out loud when in doubt.
- **Write for everyone.** Use simple, plain language. Write with accessibility and localization in mind. Avoid jargon and gendered terminology.

## Best Practices

### Screen Content
- Put the most important information first on each screen
- Format text for easy reading
- If conveying more than one idea, consider breaking text onto multiple screens
- Think about flow of information across screens

### Action-Oriented Language
- Use active voice and clear labels
- Label buttons and links with verbs
- Prefer clarity over cleverness — "Send" over "Let's do it!"
- For links, use descriptive text like "Learn more about UX Writing" instead of "Click here" (critical for screen readers)

### Language Patterns & Consistency
- Build reusable language patterns across your app for cohesion
- Adopt capitalization rules aligned with your app's style and apply consistently:
  - **Title case** = more formal
  - **Sentence case** = more casual
  - Choose a style per UI element type and use it throughout

### Possessive Pronouns
- Use sparingly — "Favorites" is better than "Your Favorites"
- If used, be consistent throughout the app and don't switch perspectives
- Avoid *we* — unclear who it refers to. "Unable to load content" is clearer than "We're having trouble loading this content"

### Multi-Step Flows
- Use "Get Started" to begin a flow
- Hint at next steps in button labels, or use "Continue" / "Next" consistently
- Use "Done" to indicate completion

### Device-Specific Writing
- Use correct gesture terms per device ("tap" for touch devices, "click" for pointer devices)
- iPhone/Apple Watch: small screens require brevity; allow personalization
- TV: content visible to multiple people; large text needed for distance viewing; keep brief

### Empty States
- Provide clear next steps and actions
- Educate people about your app; showcase your voice
- Include a button or link for next actions when possible
- Remember empty states are usually temporary — don't show crucial information that could disappear

### Error Messages
- Help people avoid errors first
- Display errors as close to the problem as possible
- Avoid blame; be clear about how to fix it
- Example: "Choose a password with at least 8 characters" is better than "That password is too short"
- Avoid interjections like "oops!" or "uh-oh" — they sound insincere
- If language alone can't address a common error, rethink the interaction

### Settings Labels
- Label settings as practically as possible
- Add explanations that describe what happens when turned on — people can infer the opposite
- Provide direct links or buttons to settings rather than describing their location

### Text Fields
- Label all fields clearly
- Use hint/placeholder text showing format (e.g., "name@example.com") or describing expected info (e.g., "Your name")
- Show errors next to the field with corrective instructions
- "Use only letters for your name" is better than "Don't use numbers or symbols"
- Avoid robotic messages like "Invalid name"

### Message Delivery
- Consider urgency, importance, and context when choosing delivery method
- Think about whether immediate action is required and how much supporting info is needed
- Use appropriate tone for the situation
- See: Notifications, Alerts, and Action Sheets guidelines
